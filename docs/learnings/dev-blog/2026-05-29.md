### What Happened Today

Big day. We shipped the **CareerLoop REST API v1** — 7 MVP endpoints, all E2E verified against real Supabase + real DeepSeek. This is the first time CareerLoop has a web-facing interface. The vision of "Telegram-first" is now permanently shifted to **web-first**, with Telegram/WhatsApp webhooks marked as permanently delayed.

### Key Accomplishments

#### 1. REST API (7 Endpoints) — Live & Verified
- **POST /v1/auth/me** — Supabase JWT auth + auto-provisioning. Every new Google user lands on `NEW_USER` state. E2E proven.
- **GET /v1/me** + **GET /v1/me/preferences** — User profile and preferences from `careerloop.users` + `user_preferences`.
- **GET /v1/briefs/latest** — TAL-style job card feed with `fit_tier` colors (emerald/amber/red), company logos (Clearbit/initials fallback), salary, location, role summary.
- **POST /v1/briefs/{brief_id}/items/{item_index}/select** — Session context persistence for coherent chat follow-up.
- **GET /v1/jobs/{job_id}** — Full job detail with company branding, match status, fit score.
- **POST /v1/jobs/{job_id}/save** + **/skip** — Swipe-right/left persistence to `user_job_relationships`.
- **POST /v1/chat/message** — Universal NL interface. NEW_USER → onboarding flow. PROFILE_READY+ → LangGraph supervisor graph (real DeepSeek).
- **POST /v1/scans** + **GET /v1/scans/{run_id}/events** (SSE) — Async scan initiation with live Server-Sent Events streaming. "Scan More" mode for forced fresh discovery. 5-min timeout, watermark-based event polling.

#### 2. SSE Scan Streaming — Major Architectural Win
The `scan_more` mode is architected with:
- **Timestamp watermark dedup** — each poll fetches only events `>= last_ts`, bounds queries to O(new events)
- **Event ID dedup** — handles same-timestamp collisions
- **Thread-safe worker pattern** — background thread with independent `DatabaseManager` connection, `DATABASE_URL` passed explicitly (not inherited from env)
- **SCANNED_EMIT_CAP** — 60 event cap to bound `run_events` rows while keeping the counter honest

#### 3. Serializers — Production-Grade Data Layer
- `company_logo()` — 3-tier fallback: explicit logo → Clearbit domain lookup → initials avatar (never broken image)
- `fit_tier()` — ≥80 strong (emerald), 60-79 good (amber), <60 weak (red)
- `description_snippet()` — max 280 chars for brief cards, 1200 for full detail
- `brief_item()` / `job_detail()` — full column mapping decoupled from v1/v2 schema drift

#### 4. Supabase JWT Auth — Universal, Self-Provisioning
- `security.py` verifies HS256 JWTs issued by Supabase
- `deps/auth.py` auto-provisions `careerloop.users` on first call with **in-process TTL cache** (300s) to avoid DB write on every request
- Same token works for web, iOS, Android — auth is transport-agnostic

#### 5. Identity Provider Layer (NEW)
- `identity_provider.py` — abstracted profile resolution: LinkedIn URL extraction from any input, CV-based identity inference, identity card generation for onboarding confirmation

### What We Pivoted

| Before | After | Rationale |
|--------|-------|-----------|
| Telegram-first delivery | **Web-first deployment** | Faster iteration cycle, no app store, no bot approval |
| WhatsApp/Meta API (planned) | **Permanently delayed** | Unnecessary complexity for current scale |
| CLI as primary interface | **API as primary interface** | Frontend consumes JSON, CLI maintained for debugging |
| Transport abstraction layer (P0) | **Reduced to P2** | Web app bypasses need for multi-transport abstraction |

### Known Issues (Tracked)

1. **companies.logo_url, domain, website are NULL** — all current companies show initials-avatar fallback. Need Clearbit backfill job.
2. **salary_min/max, jd_text, role_summary empty** for 5 real scraped jobs. Scraper doesn't populate them. Seeded test jobs have them.
3. **UV/Icron restart loop** — the scan worker thread creates a new `DatabaseManager` which might trigger schema init if `skip_schema_init` doesn't work on reconnection. Monitoring needed.
4. **In-process provision cache** — `_provisioned` dict is per-process. Multi-worker deployments will each provision independently on first call per worker.
5. **No scan timeout propagation** — 5-min SSE timeout exists but no mechanism kills a stuck scan thread.
6. **PostgresSaver checkpointer at 20%** — SQLite sessions work but Postgres checkpointing is untested with API load.
