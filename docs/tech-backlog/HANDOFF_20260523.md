## Context: What We Started With

The pipeline was fundamentally broken. `CompanyDiscoveryEngine.discover()` never hit the internet — always returned 0 companies. Root cause: `ddgs` package not installed. Every DDG-based source silently returned `[]` because `from ddgs import DDGS` threw `ModuleNotFoundError` inside a try/except. This was the single blocking bug across Phase A. Once fixed, GoogleMapsDiscovery returned 44 companies in a single test.

## Phase A — Employer Discovery

**What it does:** Finds companies that likely hire for the target role. Uses 5 DDG-powered sources.

### Fixes shipped:

**FIX 3 — Playwright `--no-sandbox`**  
All `chromium.launch()` calls were missing `args=["--no-sandbox"]`. Added to `company_discovery.py`, `ats_adapter.py`, `portal_scraper.py`, `company_portal_scraper.py`, `scrapegraph_adapter.py`.

**FIX 6 — No top-20 cap on employer discovery**  
`CompanyTargeting.top_n()` was arbitrarily capping at 20 companies. Changed signature to `top_n(n=0, min_score=50.0)` — now returns ALL companies scoring > 50, sorted descending. Candidate pool increased from `max(n*4, 100)` to 2000.

**FIX 7 — Employer dedup**  
`company_discovery.py` uses `_normalize_id(domain or name)` with a `seen` set across all 5 sources (GoogleMaps, Wellfound, Crunchbase, Inc42, YC). Prevents the same company appearing twice.

**`ddgs` + `playwright` added to `requirements.txt`**  
These were missing entirely. `ddgs>=0.1` (DuckDuckGo search) and `playwright>=1.40` added. `playwright install chromium` run.

## Phase B — Job Board Search

**What it does:** Searches 6 job boards in parallel for the target role.

### Fixes shipped:

**FIX 13 — Naukri adapter (Playwright XHR intercept)**  
Naukri API returns `{"message":"recaptcha required"}` on direct REST calls. Fixed with Playwright XHR intercept: opens `naukri.com`, intercepts `jobapi` XHR calls, extracts JSON response. Result: 10 jobs confirmed (`AI Product Engineer @ Oppdoor`, etc.).

**FIX 14 — Monster/foundit adapter**  
Old `foundit.in/middleware/jobsearch/api/v2/search` endpoint was dead (404). SRP blocks headless (304 bytes). Replaced with DDG `site:foundit.in` queries in `monster_adapter.py`.

**FIX 15 — Glassdoor adapter (`glassdoor_adapter.py` — NEW)**  
Created from scratch. Uses DDG `site:glassdoor.com/job-listing` queries. Returns 9-10 results per role/city.

**FIX 16 — Google Jobs adapter (`google_jobs_adapter.py` — NEW)**  
Created from scratch. Uses DDG queries targeting Lever/Greenhouse/Ashby indexed jobs — effectively surfaces ATS-hosted jobs via Google's index without scraping Google Jobs UI directly.

**FIX 23 — Parallel board search (ThreadPoolExecutor)**  
All 6 board sources previously ran sequentially (~14 min total). Now run via `ThreadPoolExecutor(max_workers=6)`. Wall-clock: ~14 min → ~3-4 min (dominated by slowest single source). Each worker wrapped with M1 retry.

## Phase F — Scoring

**What it does:** 14-dimension India fit scoring (salary, location, role match, company quality, etc.).

### Fixes shipped:

**FIX 21 — Score spread validated**  
Run 1 data shows compression: 61.7–74.3 range, 22/25 jobs in 60-69 band. Root cause confirmed: JobSpy/lever returns 500-char snippets → insufficient text for scorer to differentiate. Source-aware weighting applied as partial fix.

**FIX 22 — Source-aware weighting**  
`india_fit_engine.py` `_score_response_likelihood()` now weights by source quality:
- ATS direct (greenhouse/lever/ashby/workday/company_portal): `+3.0`
- Scraped (scrapegraph/naukri/foundit/instahyre/cutshort): `+1.5`
- JobSpy/LinkedIn: `+1.0`
- DDG/generic/glassdoor/google_jobs: `+0.0`

ATS-sourced results now float to the top of rankings vs snippet-only DDG results.

## Platform / Orchestration Fixes

**FIX 1 — No Anthropic dependency**  
Confirmed: zero `anthropic` or `ANTHROPIC_API_KEY` references in production code. Only test data URL (`greenhouse.io/anthropic`) in `tests.py`.

**FIX 8 — Career page ranking**  
`portal_scraper.py` `VIEW_JOBS_SELECTORS` ordered by specificity (`/careers/open` > `/jobs` > `Join Us`). First matching link wins.

**FIX 12 — Browser-first architecture**  
Portal scraper enforces L1 API intercept → L2 rendered DOM → L3 agentive → L4 raw HTML. No `requests.get()` as primary path.

**FIX 24 — Crawl cache**  
`crawl_cache.py` — 8h TTL, `expires_at` field, stale detection on read. Cache miss → full pipeline; hit → skip pipeline.

**M1 — Retry orchestration**  
`_retry(fn, retries=2, label)` static method on `OnDemandSearch`. Wraps all 6 board workers (retries=2) and all portal company scrapes (retries=1). Exponential backoff: `1.5 * (attempt + 1)` seconds.

**M2 — Source health monitoring**  
`_last_board_health` dict on `OnDemandSearch` tracks per-board result counts after every `_board_search()` call. Logged at INFO level and surfaced in `result.notes` (e.g. `"board health: ddg:3 | glassdoor:9 | google_jobs:14 | jobspy:22 | monster:5 | naukri:10"`).

**M3 — Crawl prioritisation**  
`CRAWL_STATUS_SCORE = {active:1.0, warm:0.7, pending:0.5, cold:0.2, dead:0.0}` in `company_targeting.py`. Hot/warm/cold employer prioritisation already in the scoring formula.
